Introduction:

Microchip MiWi™ protocol provides simple wireless connectivity based on IEEE 802.15.4. Microchip MiWi™ protocol is royalty free as long as implemented on Microchip PIC microcontroller and radio frequency transceiver. Please refer to the attached MiWi™ license agreement for details.

 

The MiWi™ source code is released with applications to demonstrate communications between two RF devices. The source code for each device is located in individual directories. In addition, the directory Microchip is source code for MiWi™ stack. The directory structure for MiWi™ source code is as following:

 

 

 

 

 

 

 

 

 

 

 

 


MiWi™ protocol supports Microchip PIC18, PIC24, dsPIC33 and PIC32 microcontroller families in simple C language. MiWi™ protocol can also use PIC16 microcontroller family as an end device in assembly language. For each microcontroller family, a separate project is provided under directories of MiWi Coordinator and MiWi End Device. PICDEM Z demo boards and Explorer 16 demo boards are the default development platforms. It is highly recommended to use the ZENA™ network analyzer for MiWi to catch the traffic of wireless communication over the air. It is also recommended to connect the RS232 serial port on the demo boards with a PC and use hyper terminal to monitor the output from the application and stack.

 

Required Hardware:

  To run this project, you will need two wireless nodes. Each node can be any of following setups

           

            Hardware Set 1:

                        PICDEM Z 2.4 Demo Kit (DM163027-4 OR DM163027-5)

                        OR

                        PICDEM Z Mother Board (AC163027-1)

                        PICDEM Z MRF24J40 2.4GHz Daughter Card (AC163027-4) OR

                        MRF24J40MA PICDEM™ Z 2.4GHz RF Card (AC163028)

 

            Hardware Set 2:

                        Explorer 16 (DM240001)

                        PIC24FJ128GA010 Plug-In-Module (PIM) (MA240011)

                        PICDEM Z MRF24J40 2.4GHz Daughter Card (AC163027-4) OR

                        MRF24J40MA PICtail™ Plus 2.4GHz RF Card (AC164134)

Configuring the Hardware:

 

This section describes how to set up the various configurations of hardware to run this demo.

Configuration 1:  PICDEM Z Demo Kit

Configuration 2:  Explorer 16 demo board, RF Card and PIC24FJ128GA010 or PIC32MX360F512L PIM

 

Configuration 1: PICDEM Z

1)       Connect the MRF24J40 2.4GHz RF Card to the PICDEM Z demo board

           

 

Configuration 2: Explorer 16

1)       Before attaching the PIM to the Explorer 16 board, insure that the processor selector switch (S2) is in the “PIM” position as seen in the image below.

 

2)       Short the J7 jumper to the “PIC24” setting

 

3)       Before connecting the PIM to the Explorer 16 board, remove all attached cables from both boards.  Connect the PIM to the Explorer 16 board.  Be careful when connecting the boards to insure that no pins are bent or damaged during the process.  Also insure that the PIM is not shifted in any direction and that all of the headers are properly aligned.

4)       Connect the MRF24J40 2.4GHz Daughter Card or MRF24J40MA PICTail™ Plus 2.4GHz RF Card to the first slot of edge card connector.

 

 

Firmware:

To run this project, you will need to load the corresponding firmware into the devices.  There are two methods available for loading the demos: Precompiled demos and source code projects.

 

Precompiled Demos are available in the “<Install Directory>\MiWi\MiWi Coordinator” and “<Install Directory>\MiWi\MiWi End Device” folders.  These are folders for Coordinator and End Device respectively. In each folder, one hex file for each hardware setting is provided. For how to load precompiled demos, please refer to “Getting Started – Loading a precompiled demo”.

 

The source code for this demo is available in the “<Install Directory>\MiWi\MiWi Coordinator” and “<Install Directory>\MiWi\MiWi End Device” directories.  In these directories, you will find all of the user level source and header files as well as project and workspace files for each of the hardware platforms. To compile the demo, following working environment must be established:

·         PICDEM Z Configuration: C18 v3.20 or higher

·         Explorer 16 Configuration: C30 v3.10 or higher for PIC24 and dsPIC33; C32 v1.02 or higher for PIC32.

Compile and program the demo code into the hardware platform. For more help on how to compile and program projects, please refer to the MPLAB® IDE help available through the help menu of MPLAB IDE (Help->Topics…->MPLAB IDE).

 

 

Running the Demo:

Before any demo is performed, a MiWi™ network must be formed. It is highly recommended to connect a serial cable to both demo boards and connect the ZENA sniffer hardware to monitor the operating of the network. Once the serial cable is connected to the demo board and PC, launch a hyper terminal to display the information from the demo board. The hyper terminal configuration is baud rate 19200, Data bit 8, Parity None, Stop bits 1 and Flow control None, as shown below.

 

ZENA™ network analyzer can be used to monitor the network traffic. To run ZENA™ sniffer, connect the ZENA board with PC through the USB interface, then launch ZENA software. The ZENA window will show up. Choose “MiWi™ Tools” Menu and click the menu item “Network Traffic Monitor” to launch “MiWi™ Network Monitor” window.

 

From the “MiWi™ Network Monitor” window, check “Real Time Display” box and choose proper channel. By default, this demo use channel 25. Choose “Operation” menu and click “Start Sniffing/Playback” menu item to launch the “ZENA™ Packet Sniffer” window to monitor the wireless traffic.

 

Following are the steps to setup the network with two devices:

 

a)      Power up the MiWi™ Coordinator. (PAN Coordinator)

1.      After the MiWi™ Coordinator is powered up, on the HyperTerminal, you should see the output of “Trying to find a suitable network” repeated five times and then output “Starting my own network”.

 

2.      If ZENA sniffer is used, the default channel used is channel 25. You should see four continuous Beacon Request messages.

 

b)      Power up the RFD end device.

1.      On the HyperTerminal, you should see the output of “Trying to find a suitable network”, then “Trying to join network: xx”. If the device joins the network successfully, a table of available networks and available nodes will be shown, followed by the output “Join the network successfully”. Otherwise, the end device will repeat this step until it can join a network successfully.

 

2.      If ZENA sniffer is used, you should see a Beacon Request from the end device. Then the PAN Coordinator should respond with a Beacon packet. Then RFD end device will send an Association Request; PAN Coordinator will respond with an Association Response. You should also observe continuous Data Request from the RFD end device to the PAN Coordinator.

 

c)      Demonstrate how to transmit a message

Here we demonstrate all three ways to transmit a packet in MiWi™ protocol:

      Send Report by Long Address

      Send Report by Short Address

      Send Report by Handle (Socket)

 

1.      Send Report by Long Address or Short Address

a)      Press Button 2 (RB4 on PICDEM Z demo board or RD7 on Explorer 16 demo board) on any of the board. A packet will be sent to the other board to toggle LED 2 (RA1 on PICDEM Z demo board or D9 on Explorer 16 demo board).

b)      In this step, we try to send report by long address and short address alternatively.

c)      If the hyper terminal is used, you should see the output on the screen “Send Report by Long Address” and “Send Report by Short Address” displayed alternatively.

 

d)      If the ZENA sniffer is used, you should be able to see a packet send out to toggle the LED from one board to the other.

Note: Even though we send out packet in different ways in application layer, the MiWi stack is programmed to choose the way with least cost in the MAC layer. As the result, you may see that the node sends out identical packets even though the application layer sends out packet in differently ways.

 

2.      Demonstrate the Cluster Socket.

a)      Press Button 1 (RB5 on PICDEM Z demo board or RD6 on Explorer 16 demo board) of RFD and then Button 1 (RB5 on PICDEM Z demo board or RD6 on Explorer 16 demo board) of Coordinator within one second. LED 1 (RA0 on PICDEM Z demo board or D10 on Explorer 16 demo board) on both boards should turn on if the operation is successful.

b)      In this step, we try to establish a cluster socket between two devices. For details of cluster socket, please refer to section “CLUSTER SOCKET” in application note AN1066 “MiWi™ Wireless Networking Protocol Stack”.

c)      If the hyper terminal is used, you should see the output on the screen “Opening Cluster Socket” and then followed by the status of the open socket operation: “Found a socket: XX” if successful, or “socket request failed” otherwise.

 

d)      If ZENA sniffer is used, you should be able to see that two devices exchange Cluster Socket Request (Report ID 0x10) and Cluster Socket Response (Report ID 0x11) to establish socket connection.

 

e)      After the cluster socket has been established, by pressing Button 2 (RB4 on PICDEM Z demo board or RD7 on Explorer 16 demo board) on one board, LED 2 (RA1 on PICDEM Z demo board or D9 on Explorer 16 demo board) should be toggled on the other board.

a.       In this step, we try to send a message by way of socket.

b.      If hyper terminal is used, you should be able to see output “Send Report by Handle(Socket)”.

c.       If ZENA sniffer is used, you should be able to see packet transmitted to toggle the light.

 

d.      To get out of socket mode and send report by long address or short address, you may press Button 1 (RB5 on PICDEM Z demo board or RD6 on Explorer 16 demo board) of the board and let the socket seeking procedure times out. Once socket request procedure fails, pressing Button 2 (RB4 on PICDEM Z demo board or RD7 on Explorer 16 demo board) will send report by long address or short address alternatively.

 

 

Trouble Shooting:

 

Index

Symptom

Solution

1

No traffic on ZENA

1.      Check if ZENA operates at the proper channel. The firmware uses channel 25 by default.

2.      Check if power supply is OK. Check if battery switch is ON when 9V battery is used. Sometimes, the connector for 9V battery may be loose.

3.      Press MCLR button to reset the device.

2

Packet displayed in ZENA does not interpolate packet correctly.

Check if the Packet Sniffer window is for the correct protocol. ZENA supports ZigBee™, MiWi™ and MiWi™ P2P stacks. Be sure MiWi™ is used in this demo.

3

Traffic stopped after a few packets

Reset the Coordinator or the RFD device by pressing MCLR button.

4

There is a time lag between the Button 2 (RB4 on PICDEM Z demo board or RD7 on Explorer 16 demo board) of Coordinator pressed down and LED 2 (RA1 on PICDEM Z demo board or D9 on Explorer 16 demo board) of RFD device toggled.

That is expected. RFD device can only receive a packet when it wakes up and ask for data from its parent. The maximum time lag is the polling rate of the RFD device. In this demo, it is about 1 second.

5

I continuously press the button, but the LED on the other board does not toggle.

To avoid triggering false button press signal, the firmware defines debounce time. Continuously press the button within debounce time will be discarded.

6

The RF range between two devices is limited to less than 10 meters.

Your devices may be working on a noisy channel. Try to change definition of ALLOWED_CHANNELS to a different channel for both Coordinator and RFD devices, compile and reprogram both devices.
